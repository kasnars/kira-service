# 用户模块（增删改查）整体逻辑说明

## 一、项目分层架构

NestJS 采用的是**分层架构**，一个请求会依次经过以下层级：

```
前端请求
  ↓
Controller（控制器层）  ← 负责接收请求、取参数、返回结果
  ↓
Service（服务层）        ← 负责业务逻辑（加密、校验、组装数据）
  ↓
Repository（数据访问层） ← 负责和数据库交互（增删改查）
  ↓
数据库（SQLite 文件）
```

**为什么要分层？** 每层只做自己的事，职责清晰。比如 Controller 不应该知道密码要加密，Service 不应该知道 HTTP 状态码是什么。

## 三、一个完整请求的流程（以"创建用户"为例）

### 第 1 步：前端发送请求

```javascript
// 前端代码
fetch('http://localhost:3000/users', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    username: 'admin',
    password: '123456',
    nickname: '管理员',
  }),
});
```

### 第 2 步：NestJS 收到请求，路由匹配到 Controller

`main.ts` 中配置了全局 `ValidationPipe`，请求会先经过校验管道。

NestJS 根据请求方法（POST）和路径（/users），匹配到 `user.controller.ts` 中：

```typescript
@Controller('users')     // 匹配路径 /users
export class UserController {
  @Post()                // 匹配 POST 方法
  create(@Body() dto: CreateUserDto) { ... }
}
```

### 第 3 步：ValidationPipe 校验请求参数

`@Body() dto: CreateUserDto` 这一步做了两件事：

1. **解析**：把请求体 JSON 解析成 `CreateUserDto` 对象
2. **校验**：根据 `CreateUserDto` 中声明的规则检查每个字段：

```typescript
// create-user.dto.ts
@IsNotEmpty({ message: '账号不能为空' })
@IsString({ message: '账号必须是字符串' })
username: string;

@MinLength(6, { message: '密码至少 6 位' })
password: string;
```

- 如果校验失败 → 直接返回 400 错误，**请求不会继续往下走**
- 如果校验通过 → 进入下一步

### 第 4 步：Controller 调用 Service

校验通过后，Controller 拿到干净的 `dto` 对象，调用 Service：

```typescript
// user.controller.ts
@Post()
create(@Body() dto: CreateUserDto) {
  return this.userService.create(dto);  // 转发给 Service 处理
}
```

**注意：Controller 这里只做"转发"，不写任何业务逻辑。**

### 第 5 步：Service 处理业务逻辑

```typescript
// user.service.ts
async create(dto: CreateUserDto) {
  // 1. 密码加密：明文不能存数据库
  const hashedPassword = await bcrypt.hash(dto.password, 10);

  // 2. 创建实体对象（此时还没写入数据库）
  const user = this.userRepo.create({
    ...dto,
    password: hashedPassword,
  });

  // 3. 真正保存到数据库（执行 SQL INSERT）
  await this.userRepo.save(user);

  // 4. 返回创建后的用户信息（不返回密码）
  return this.findOne(user.id);
}
```

这里的关键点：
- `this.userRepo.create()` → 只是在内存中创建一个对象，**不操作数据库**
- `this.userRepo.save()` → 才真正执行 INSERT 写入数据库
- 分开写的好处：可以在 save 之前做更多处理（比如检查用户名是否存在）

### 第 6 步：Repository 执行数据库操作

`this.userRepo.save(user)` 底层执行的 SQL：

```sql
INSERT INTO users (username, password, nickname, createdAt, updatedAt)
VALUES ('admin', '$2b$10$xxx...', '管理员', '2026-05-20 22:44:19', '2026-05-20 22:44:19');
```

这个 SQL 是 TypeORM 自动生成的，你不需要手写。

### 第 7 步：逐层返回结果

```
数据库返回新用户的 id → Service.findOne(1) 查询 →
Controller 拿到结果 → NestJS 序列化成 JSON → 前端收到响应
```

前端收到的响应：

```json
{
  "id": 1,
  "username": "admin",
  "nickname": "管理员",
  "createdAt": "2026-05-20T14:44:19.000Z",
  "updatedAt": "2026-05-20T14:44:19.000Z"
}
```

注意：**密码字段没有返回**，因为在 `findOne` 中用 `select` 排除了它。

## 四、其他操作的流程对比

### 查询所有（GET /users）

```
前端 GET /users
  → Controller.findAll()           匹配 @Get()
    → Service.findAll()
      → userRepo.find()            执行 SELECT * FROM users
        → 返回所有用户列表（不含密码）
```

### 查询单个（GET /users/1）

```
前端 GET /users/1
  → Controller.findOne(@Param('id') id)    :id 匹配 URL 中的 1
    → Service.findOne(1)
      → userRepo.findOne({ where: { id } })  执行 SELECT * FROM users WHERE id = 1
        → 返回单个用户（不含密码）
```

### 更新（PUT /users/1）

```
前端 PUT /users/1, body: { nickname: '新昵称' }
  → Controller.update(id, @Body() dto)
    → ValidationPipe 校验（UpdateUserDto 字段都是可选的）
    → Service.update(1, dto)
      → 如果 dto.password 存在，加密
      → userRepo.update(1, dto)     执行 UPDATE users SET nickname = '新昵称' WHERE id = 1
        → 返回更新后的用户数据
```

### 删除（DELETE /users/1）

```
前端 DELETE /users/1
  → Controller.remove(@Param('id') id)
    → Service.remove(1)
      → userRepo.delete(1)          执行 DELETE FROM users WHERE id = 1
        → 无返回值（204 状态码）
```

## 五、核心概念总结

### 1. 依赖注入（DI）

NestJS 自动管理类之间的依赖关系：

```typescript
// 你不需要这样写：
const userService = new UserService(new UserRepository());

// NestJS 帮你自动注入：
constructor(private readonly userService: UserService) {}  // 直接声明类型就行
```

类比 Vue：类似 `provide / inject`，父组件 provide 一个值，子组件 inject 就能用。

### 2. 装饰器（Decorator）

装饰器是 TypeScript/JavaScript 的语法，用 `@` 符号声明，给类或方法"贴标签"：

| 装饰器 | 作用 | 类比 |
|--------|------|------|
| `@Controller('users')` | 标记这个类处理 /users 路由 | Vue Router 的路由配置 |
| `@Get()` / `@Post()` | 标记这个方法处理 GET/POST 请求 | 后端的路由定义 |
| `@Injectable()` | 标记这个类可被注入 | Vue 的 composable 函数 |
| `@Column()` | 标记这个属性是数据库列 | Sequelize / Prisma 的字段定义 |
| `@IsNotEmpty()` | 标记这个字段不能为空 | 表单校验规则 |

### 3. DTO 的作用

DTO 是 Controller 层的"门卫"，负责检查请求参数是否合法：

```
前端请求 → DTO 校验 → 通过则进 Service，不通过则直接返回 400
```

好处：
- Service 层拿到的数据一定是干净的，不用再做 `if (dto.username)` 这种判断
- 校验规则集中管理，一眼就能看出接口需要什么参数

### 4. Entity 与数据库的关系

```typescript
// Entity 定义
@Entity('users')
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ unique: true })
  username: string;
}
```

TypeORM 会自动把它翻译成 SQL 建表语句：

```sql
CREATE TABLE users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username VARCHAR UNIQUE,
  password VARCHAR,
  nickname VARCHAR,
  createdAt DATETIME DEFAULT CURRENT_TIMESTAMP,
  updatedAt DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

这就是 `synchronize: true` 的作用 —— 开发时自动同步表结构。

## 七、调试技巧

当接口报错时，看**终端日志**（不是浏览器控制台）：

```
[Nest] 12345 - Nest application successfully started    ← 服务器启动成功
[Nest] 12345 - Mapped {/users, POST} route               ← 路由注册成功
[Nest] 12345 - ValidationPipe: BadRequestException       ← DTO 校验失败
[Nest] 12345 - QueryFailedError                          ← 数据库操作失败
```

用 `curl` 或 Postman 测试接口：

```bash
# 创建用户
curl -X POST http://localhost:3000/users \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"123456","nickname":"测试"}'

# 查询列表
curl http://localhost:3000/users

# 更新用户
curl -X PUT http://localhost:3000/users/1 \
  -H "Content-Type: application/json" \
  -d '{"nickname":"新名字"}'

# 删除用户
curl -X DELETE http://localhost:3000/users/1
```
